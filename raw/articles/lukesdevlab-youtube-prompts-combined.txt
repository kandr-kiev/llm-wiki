These basic spelling errors may seem familiar. LLMs, the kind of artificial intelligence that powers chatbots and other text-generators, are not built to understand spelling. It’s been a running joke for years that whenever a company unveils a new AI model, you should ask it how many ‘r’s are in the word strawberry. As researchers have previously explained when we’ve asked about these spelling conundrums, AI doesn’t perceive sentences as units of language made up of words and letters. Many LLMs are built on transformers models, which break down text into tokens, which can be full words, syllables, or letters, depending on the model. Instead of “reading” like a human would, the AI converts the text into numerical representations of itself, which are then contextualized to help the AI come up with a logical response.

import bpy
import os

EXPORT_ROOT = "/absolute/path/to/exports/marble_run"  # must exist or be created
os.makedirs(EXPORT_ROOT, exist_ok=True)

def export_selected_glb(obj_name: str, filename: str):
    obj = bpy.data.objects.get(obj_name)
    if obj is None:
        raise RuntimeError(f"Missing object: {obj_name}")
    bpy.ops.object.select_all(action='DESELECT')
    obj.select_set(True)
    bpy.context.view_layer.objects.active = obj
    filepath = os.path.join(EXPORT_ROOT, filename)
    bpy.ops.export_scene.gltf(
        filepath=filepath,
        export_format='GLB',
        use_selection=True,
        export_apply=True,
        export_materials='EXPORT',
    )
    return filepath

### Verlet Physics (core implementation — no built-in soft-body plugin):

Each rope particle `i` has `pos`, `prev_pos`, and optional `pinned: bool`.

1. **Integration (per sub-step):**
   ```
   velocity = pos - prev_pos
   prev_pos = pos
   pos += velocity + gravity * dt * dt
   ```
   Gravity: `(0, 980)` px/s². Use **8 sub-steps** per frame (`dt / 8`).

2. **Distance constraints** between consecutive particles on the same rope:
   - Rest length = total rope length / (particle_count - 1)
   - Relax each constraint **5 iterations** per sub-step using symmetric correction (move both particles unless pinned).

3. **Pin constraints:** Pinned particles have `pos` and `prev_pos` locked to anchor each sub-step.

4. **Cliff collision:** Particles and boulder cannot penetrate cliff `StaticBody2D` rectangles. Use Godot physics for boulder-cliff; rope particles may use simple AABB clamp against cliff edges.

5. **Boulder ↔ rope interaction (required):**
   - Each frame, for each rope particle within boulder radius + margin: push particle outward along normal from boulder center, and apply equal-and-opposite impulse to boulder velocity (approximate point collision).
   - Without this, the boulder will ghost through ropes — that is a failure.

### Critical Implementation Rules (coding traps):
- **Sub-stepping is mandatory.** Single-step Verlet with large dt will explode constraints.
- **Constraint iteration count matters.** Fewer than 3 iterations produces rubber-band bridges; document your iteration count.
- **Do not use `RigidBody2D` joints for the rope itself** — the rope must be custom Verlet + constraints (joints optional only for boulder if needed).
- **Anti-tunneling:** If boulder speed × dt > rope particle spacing, do 2 boulder collision passes per sub-step.
- **BUILD/TEST state machine** must fully reset particle velocities when entering TEST (zero initial rope wobble except gravity).
