# Інтеграція Graphify з LLM-WIKI: стратегія, теорія, техніка, рекомендації
Цей документ є оновленою версією стратегії інтеграції Graphify з локальною баizoю LLM-WIKI. Він розділений на три частини: **теоретичну**, **технічну** та **інтеграційну**. Задум — розмежувати два шари: Graphify буде будувати структурну карту проєкту, а LLM-WIKI залишатиметься людині-читабельною кураторською базою знань.
## 1. Теоретична частина
### 1.1 Що таке Graphify у контексті бази знань
Graphify — окремий інструмент з відкритим кодом (MIT), який перетворює дерево файлів проєкту на queryable knowledge graph. У ньому є три фундаментальні рівні представлення:
- **ноди** — файли, функції, класи, сутності, концепти;
- **ребра** — відносини: виклики, імпорти, посилання, семантичні зв'язки;
- **ком'юніті** — групи взаємопов'язаних вузлів, отримані через Leiden-алгоритм без векторних embeddingів.
Graphify не замінює LLM-WIKI, а дає їй новий вихід: не лише людино-читабельні сторінки, але й структурування проєкту залежностей, яку можна інжестувати додатковим raw-документом і потім перетворити на concept-сторінку в wiki.
### 1.2 Чому граф кращий за прості індекси для коду
Поточна LLM-WIKI-стратегія полягає в ручному інжестуванні сирців, оформленні їх за SCHEMA.md і побудові сторінок сутностей/концептів. Це відмінно працює для тексту, але має недоліки для коду:
- контекстне вікно не розширюється за допомогою реальної карти викликів;
- важко автоматизувати оновлення, коли змінюється архітектура;
- аналіз залежностей вимагає повторного читання файлів.
Graphify вирішує це саме структурним рівнем: tree-sitter AST дає точні ребра без викликів LLM, а семантичний прохід по доках/PDF/зображеннях дає концептуальні зв'язки між модулями. На виході не має embedding-vectorstore — є точний граф, який можна інdexовано перевіряти.
### 1.3 Теорія форматів виходу
Graphify генерує три артефакти в `graphify-out/`:
- `graph.json` — повний persistent graph у форматі, сумісному з MCP/agent-запитами;
- `graph.html` — інтерактивний візуальний граф у браузері;
- `GRAPH_REPORT.md` — зведення: god-nodes, несподівані зв'язки, запропоновані питання.
Кожне ребро має тег:
- `EXTRACTED` — явно визначено у джерелі;
- `INFERRED` — отримано через семантичний аналіз, з confidence від `0.4` до `0.9`.
Для wiki важливий саме `GRAPH_REPORT.md`: він містить людині-читабельний опис, а не сирий JSON. Це дозволяє використовувати його як dając source для concept-сторінки, не вводячи довільного тексту.
### 1.4 Як Graphify співіснує з LLM-WIKI, а не конфліктує з нею
Це два різних шари, і конфлікт виникає лише за однієї умови:Graphify-артефакти всередині `wiki/`. Потрібно тримати:
- `graphify-out/` — окремо від `llm-wiki/`, в корені проєкту;
- `llm-wiki/raw/articles/` — для інжестування сирового `GRAPH_REPORT.md` або його фрагментів;

## 2. Технічна частина
### 2.1 Встановлення
**Вимоги:**
- Python 3.10+
- Пакет PyPI: `graphifyy` (double-y), не переплутайте з одно-Y-варіантом — це інший пакет.
- CLI команда: `graphify`
- Для Hermes: `graphify install --platform hermes` або `--project` для скоупу на проєкт.
**Рекомендації:**
- Краще використовувати `uv tool install graphifyy`, але одразу додати `uv tool update-shell`, щоб `graphify` з'явився у PATH. Без цього CLI може бути невидимим.
- Альтернатива: `pipx install graphifyy` або `pip install graphifyy --break-system-packages` в ізольованому venv.
- Hermes-скіл встановлюється в профіль чи в `.agents/skills/graphify/` при `--project`. Треба переконатися, що ця дія не конфліктує з вашим workflow через `skill_manage`, `Hermes`.
### 2.2 Конфігурація Hermes та LLM-бекенд
Graphify для коду не викликає LLM: tree-sitter AST парситься локально і детерміновано. LLM потрібен лише для семантичного проходу по markdown, PDF, зображеннях, відео. За замовчуванням Graphify використовує **той самий AI-клієнт, що налаштований у вашому асистенті** — тобто для Hermes це не обов'язково http://localhost:1234/v1.
Тому перед запуском на широкому корпусі:
1. Перевірте `graphify config --help`, щоб дізнатись точну назву змінної/параметра для моделі.
2. Якщо використовуєте локальний сервер через `llama-swap` або LM Studio, вкажіть `OPENAI_BASE_URL` або відповідний параметр.
3. Переконайтеся, що Hermes не конфліктує з Graphify за модель: Hermes використовує налаштованого провайдера, а Graphify — власний конфіг для семантичного проходу.
Важливо: raw-код НЕ виходить за межі машини. На LLM йдуть лише семантичні дескриптори документів.
### 2.3 Запуск
```bash
cd /шлях/до/проєкту
graphify .
```
Вихідні файли:
- `graphify-out/graph.html` — відкрийте в браузері;
- `graphify-out/graph.json` — queryable graph, persistent;
- `graphify-out/GRAPH_REPORT.md` — зведення;
- `graphify-out/cache/` — інкрементальний кеш для повторних запусків.
Тегування ребер:
- `EXTRACTED` — явно взято з коду/документа;
- `INFERRED` — згенеровано через семантичний аналіз.
### 2.4 Інтеграція з Hermes
- `graphify install --platform hermes` — реєстрація глобально;
- `graphify install --project --platform hermes` — реєстрація в проєкті, файли пишуться в `.agents/skills/graphify/`;
- CLI оновлюється кожні 1-2 тижні, тому перед використанням завжди перевіряйте `graphify --help`, щоб не отримати зміну флагів.
### 2.5 Автооновлення
- Після коміту: `graphify hook install` для git hooks;
- Потім Graphify перебудовує тільки змінені файли завдяки `cache/`;
- Не комітьте `graphify-out/` в репозиторій.
