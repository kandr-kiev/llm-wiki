⌨️ **crates.io: development update**
Минуло ще півроку після останнього оновлення розробки, і команда crates.io була зайнята.
Ось короткий виклад найпомітніших змін і вдосконалень, внесених у crates.io з тих пір.
Облікові записи Crates.io завжди були тісно пов’язані з GitHub: вхід означає «Увійти за допомогою GitHub», а ваш ідентифікатор crates.io — це ваше ім’я користувача GitHub.
Сторінки ящиків таких ящиків тепер показують дружній банер «Ця залежність може вам не знадобитися», що описує стандартну заміну бібліотеки, а замінені ящики в списках залежностей отримують маленьку піктограму лампочки з подібною підказкою.
Набір даних, що стоїть за цією функцією, знаходиться в новому сховищі даних rust-lang/std-replacement-data разом із задокументованою політикою включення: лише стандартні заміни бібліотек, кожен запис має цитувати стабільний std, core або alloc API та версію Rust, а супроводжувачі ящиків отримують вікно сповіщень і коментарів перед додаванням запису.
📎 https://blog.rust-lang.org/2026/07/13/crates-io-development-update/

⌨️ **Launching the Rust Foundation Maintainers Fund**
Якщо ви хочете фінансово підтримати розвиток Rust, будь ласка, зробіть пожертву до Фонду підтримки Rust Foundation.
Кілька місяців тому фонд Rust Foundation оголосив про заснування фонду підтримувачів фонду Rust Foundation (RFMF).
Відтоді проект Rust тісно співпрацює з Rust Foundation, щоб визначити, як саме цей фонд буде використовуватися для підтримки супроводжуючих Rust.
Основна мета команди фінансування полягає в тому, щоб супроводжувачі, які працюють над Rust і його інструментальним набором, отримували належну підтримку.
Команда фінансування обере Maintainers in Residence на основі наявності фінансування та потреб у обслуговуванні в рамках проекту Rust і допоможе забезпечити їх успіх.

Хоча це не дозволить змінювати наявні файли випуску на місці, це дозволить зловмиснику змінити URL-адреси, надані на python.org для кожного файлу випуску, включаючи URL-адреси матеріалів перевірки.
Однак через давність уразливості (існує в кодовій базі з 2014 року) ми не маємо абсолютної впевненості, крім наших журналів і резервних копій бази даних.
Ми вважаємо, що спроби використати цю вразливість були б «гучними» та були б швидко виявлені, враховуючи кількість подальших інструментів і дистрибутивів, які автоматично перевіряють матеріали Sigstore та PGP.

⌨️ **Packaging Council Inaugural Election Dates**
Після нещодавнього затвердження PEP 772 – процесу управління Пакувальною радою, створюється нова Python Packaging Council (PPC) із широкими повноваженнями щодо специфікацій пакування та повноваженнями координувати роботу з пакування Python.
Вибори першої PPC проходитимуть паралельно з виборами ради Python Software Foundation (PSF) у 2026 році.
Рада також слугуватиме координаційним органом для екосистеми упаковки Python, співпрацюючи з багатьма зацікавленими сторонами з широкої спільноти Python над постійно покращуючим користувальницьким досвідом упаковки.
Це включатиме супроводжувачів різних інструментів пакування, таких як Python Packaging Authority (PyPA), основна команда Python, Керівна рада Python і PSF.
Два кандидати, які набрали найбільшу кількість голосів, будуть призначені Когортою А з дворічним терміном, а три кандидати, які отримають наступну найбільшу кількість голосів, будуть призначені Когортою B з однорічним терміном.
