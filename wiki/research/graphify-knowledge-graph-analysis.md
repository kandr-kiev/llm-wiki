# Graphify та Knowledge Graph у LLM-WIKI: Аналіз Архітектури

## 1. Поточна Архітектура

### 1.1 Два Різні Графи

| Граф | Тулза | Джерело | Вихід | Призначення |
|---|---|---|---|---|
| **Wiki Graph** | `wiki_graph_generator.py` | `wiki/` (wikilinks) | `graph-from-wiki.json` | Зв'язки між темами wiki |
| **Code Graph** | Graphify CLI | Код (tree-sitter AST) | `graphify-out/graph.json` | Залежності коду |

### 1.2 Wiki Graph (`wiki_graph_generator.py`)

**Алгоритм:**
1. Scan всіх wiki `.md` файлів
2. Extract `]` посилань з контенту
3. Створити nodes (кожна wiki сторінка) + edges (wikilinks)
4. Output: `graph-from-wiki.json` (NetworkX-сумісний формат)

**Вихід:**
- Nodes: 1485 (кожна wiki сторінка)
- Edges: 6280 (wikilinks між сторінками)
- Communities: 16 (тегові спільноти)

**Формат graph.json:**
```json
{
  "metadata": {
    "generator": "wiki_graph_generator.py",
    "version": "1.0.0",
    "generated_at": "2026-07-23T14:38:00Z",
    "source": "wiki/ (wikilinks)",
    "total_pages": 1485
  },
  "nodes": [
    {
      "id": "slug",
      "label": "Title",
      "type": "comparison",
      "tags": ["tag1", "tag2"],
      "confidence": 0.7
    }
  ],
  "edges": [
    {
      "source": "slug1",
      "target": "slug2",
      "type": "wikilink",
      "weight": 1
    }
  ],
  "communities": [
    {"id": "community_tag", "label": "tag", "nodes": [...], "size": 10}
  ]
}
```

### 1.3 Graphify Bridge (`graphify_bridge.py`)

**Призначення:** Міст між `graphify-out/graph.json` та `wiki/`

**Алгоритм:**
1. Завантажити `graph.json` → extract nodes + edges
2. Завантажити `wiki/index.md` → extract all wiki slugs
3. Fuzzy match: graph node labels ↔ wiki slugs
4. Для кожного match:
   - Додати wikilink `]` до wiki сторінки якщо не існує
   - Додати graph node як "related_structure" у frontmatter
5. Знайти orphan graph nodes → запропонувати нові wiki сторінки

**Поточний стан:** Працює, але очікує `graph.json` від Graphify CLI (не від wiki_graph_generator).

### 1.4 Integration Plan (`graphify-llm-wiki-integration-plan.md`)

**Філософія:** Graphify — це детермінований картограф, LLM-WIKI — це кураторський архів. Перший каже що із чим пов'язано, другий пояснює чому це важливо.

## 2. Як Планувалося

### 2.1 Workflow за Integration Plan

```
1. Збір сировини: Graphify будує граф → graphify-out/
2. Вибірковий експорт: беремо GRAPH_REPORT.md, відфільтрований фрагмент
3. Інжестування: запис у raw/articles/graphify-<проєкт>-<дата>.md
4. Концептуалізація: Монті/Архівіст пише concept-сторінку в wiki/concepts/
5. Сторонкові entity: створюються entity-сторінки з wikilinks
6. Верифікація: перевірка broken links в wiki, оновлення index.md
7. Повторюваність: graphify hook install для автоперебудови при комітах
```

### 2.2 Роль Graphify CLI

Graphify CLI — це окремий інструмент з відкритим кодом (MIT), який перетворює дерево файлів проєкту на queryable knowledge graph.

**Три фундаментальні рівні:**
- **Ноди** — файли, функції, класи, сутності, концепти
- **Ребра** — відносини: виклики, імпорти, посилання, семантичні зв'язки
- **Ком'юніті** — групи взаємопов'язаних вузлів (Leiden-алгоритм без embeddingів)

**Для чого потрібен:**
- Tree-sitter AST дає точні ребра без викликів LLM
- Семантичний прохід по docs/PDF/зображеннях дає концептуальні зв'язки між модулями
- На виході — точний граф, який можна індексовано перевіряти

## 3. Відомі Практики Впровадження

### 3.1 Microsoft GraphRAG

**Архітектура:**
- Створює knowledge graph на основі вхідного корпусу текстів
- Community summaries + graph ML outputs для prompt augmentation при query
- Два типи пошуку: Local (деталі) та Global (загальні питання)

**Переваги:**
- Substantial improvement в answering complex questions
- Hierarchical approach до RAG
- Open source (GitHub)

**Недоліки:**
- Вимагає LLM для extraction (дорого)
- Повільно для великих корпусів
- Складна настройка

### 3.2 Neo4j GraphRAG

**Архітектура:**
- Графова база даних Neo4j + LLM
- Hybrid retrieval: vector similarity + graph traversal (RRF)
- Multi-granular matching: entities, chunks, relations

**Переваги:**
- Production-ready
- Масштабований
- GRC (GraphRAG: The Definitive Guide for O'Reilly)

**Недоліки:**
- Вимагає графової БД
- Складна інтеграція
- Високі вимоги до інфраструктури

### 3.3 Wikilinks + Knowledge Graph

**Принцип:** Markdown файли + wikilinks = knowledge graph

**Переваги:**
- Простота
- Людино-читабельний формат
- Git-friendly
- Не потребує БД

**Недоліки:**
- Немає typed relationships (тільки implicit links)
- Немає ontology (folksonomy)
- Важко автоматизувати traversal
- Немає confidence scores

| Недолік | Вплив | Мітигація |

### 5.3 Wikilinks + Knowledge Graph

| Недолік | Вплив | Мітигація |
|---|---|---|
| Немає typed relationships | Невизначені зв'язки | Named edges (predicates) |
| Немає ontology | Folksonomy chaos | Vocabulary curation |

## 7. Висновки

**Graphify CLI НЕ замінює `wiki_graph_generator.py`** — вони роблять зовсім різні речі:
- Graphify CLI аналізує **код** (залежності, виклики, структуру)
- `wiki_graph_generator.py` аналізує **wiki** (wikilinks, спільноти)

**Обидва потрібні** для повного knowledge graph системи.

**Wiki Graph** — це folksonomy (теги, wikilinks), **Code Graph** — це ontology (typed edges, tree-sitter AST).

**Bridge** (`graphify_bridge.py`) — це міст між ними, який допомагає синхронізувати зв'язки.

**Висновок:** Наша архітектура правильна. Потрібно лише встановити Graphify CLI для code graph.
